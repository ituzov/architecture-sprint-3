# Задание 1. Анализ и планирование

### 1. Описание функциональности монолитного приложения

#### Управление отоплением
- **Пользователи** могут удалённо включать или выключать отопление в своих домах.
- **Система** поддерживает выполнение команд управления отоплением, поступающих из веб-интерфейса.

#### Мониторинг температуры
- **Пользователи** могут просматривать актуальную температуру в своих домах через веб-интерфейс.
- **Система** получает данные о температуре с установленных в домах датчиков и сохраняет их в базе данных.

---

### 2. Анализ архитектуры монолитного приложения

- **Язык:** Java.
- **База данных:** PostgreSQL.
- **Архитектура:** Монолит — все компоненты (обработка запросов, бизнес-логика, доступ к данным) собраны в одном приложении.
- **Взаимодействие:** Синхронное, запросы обрабатываются последовательно.
- **Масштабируемость:** Ограничена, так как невозможно масштабировать отдельные компоненты.
- **Развёртывание:** Не оптимизировано — для обновлений приходится останавливать весь монолит.

---

### 3. Определение доменов и границ контекстов

#### Домен «Управление устройствами» (Device Management)
- **Описание:** Включает функционал дистанционного управления отоплением, настройки оборудования и контроль статусов устройств.
- **Граница контекста:** Взаимодействие с физическими устройствами (например, отопительными котлами, умными термостатами) и логика принятия решений по управлению.

#### Домен «Мониторинг температуры» (Temperature Monitoring)
- **Описание:** Получение, валидация, хранение и отображение данных от температурных датчиков.
- **Граница контекста:** Датчики, данные в реальном времени и история измерений.

#### Домен «Управление пользователями и доступом» (User Management & Access Control)
- **Описание:** Регистрация, аутентификация, управление правами доступа (например, проверка, что пользователь видит только свой дом).
- **Граница контекста:** Все аспекты безопасности, связанные с аутентификацией и авторизацией.

---

### 4. Проблемы монолитного решения

- **Сложность масштабирования:** При увеличении нагрузки невозможно масштабировать отдельные компоненты, приходится масштабировать весь монолит.
- **Ограничения по скорости разработки:** Команде приходится работать над общим кодом, что приводит к конфликтам и замедлению разработки.
- **Частые деплои:** Обновления требуют остановки всего приложения, что недопустимо для системы с круглосуточной работой.
- **Высокая когнитивная нагрузка:** Новым разработчикам сложно разобраться в большом объёме кода.
- **Зависимость между модулями:** Любое изменение может повлиять на работу других компонентов.

## ✅ Задание 4. Создание и документирование API

### 1. Тип API

В системе используются два типа API:

- **REST API (OpenAPI/Swagger)** — используется для общения между API Gateway и микросервисами.  
  Он понятен, поддерживается всеми клиентами, легко документируется и масштабируется.

- **Event-driven API (AsyncAPI)** — используется для взаимодействия микросервисов между собой через Message Bus (RabbitMQ / Kafka).  
  Это позволяет реализовать событийно-ориентированную архитектуру с низкой связностью и высокой гибкостью.
